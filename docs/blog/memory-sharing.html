Your Project's Memory, Everywhere It Runs

MoFlo learns about your project and remembers it across sessions — but the hard-won part of that memory used to be trapped wherever it was first learned. A new git worktree, a fresh machine, a teammate's clone: each started blank on everything you'd taught the project. Over the last few days we closed that gap. The headline: across git worktrees, your MoFlo learnings now follow you automatically, with no setup at all — and for teams and multiple machines, one guided command wires up the rest. Here's what changed, and why it's safe.

The premise: MoFlo remembers, until it doesn't

The whole pitch for MoFlo is that an assistant which remembers beats one that starts cold every session. As you work, it accumulates durable knowledge about your project — the gotcha that cost you two hours, the decision you'll regret re-litigating next month, the pattern that turned out to be the right one. That lives in a local SQLite database at .moflo/moflo.db, and it's what makes the next session smarter than the last.

There's a catch. That database is gitignored — deliberately, because most of what's in it is machine-local scratch. Step outside the one checkout where you built up that knowledge and you don't land completely cold: MoFlo reindexes your code and rebuilds its structural model on its own. But that rebuild costs minutes on a large repo — and the durable, hard-won learnings don't come back with it. Spin up a git worktree to review a PR? It re-indexes the code, but your learnings are gone. Clone the repo on your laptop instead of your desktop? Same. Onboard a teammate? They inherit none of what you already taught the project, and re-learn it the hard way.

The knowledge was real, durable, and hard-won — and it was stranded. That's the gap these changes close.

The one insight that makes it safe

The naive fix — "just point every checkout at one shared database" — is a trap, and understanding why is the key to everything else. So before the solution, the crux.

Not everything in that database is worth sharing. It holds three very different kinds of data:

Kind Examples Share it?
Durable learnings, knowledge — things you or Claude decided are worth keeping Yes — this is the good stuff
Structural the code map, guidance chunks, test index — a searchable model of this branch's code No — it's branch-specific and rebuilds cheaply
Ephemeral task lists, swarm run-state, epic state No — it's per-run scratch, meaningless elsewhere

Only the durable slice — the learnings and knowledge namespaces — is user-authored, slow to recreate, and identical no matter which checkout you're in. The structural data is actively harmful to share: a code map from one branch is wrong on another, and syncing it just means stale search results after a checkout. So the rule the whole design is built on is simple:

Share only the durable slice. Let everything else stay local and rebuild itself.

There's a second, subtler reason not to live-share the whole database, and it's the one that bites silently. MoFlo's semantic search is powered by an in-memory vector index (HNSW). Each running MoFlo daemon builds its own copy of that index. If two daemons pointed at one shared database file, a learning written by the first would never show up in the second's index — the row is in the file, but search can't see it until a full rebuild. You'd get a database that looks shared but a search that quietly lies. Sharing only the small, append-mostly durable slice sidesteps that entirely.

That one distinction — durable travels, structural stays — is the foundation under everything below.

Worktrees just work — automatically

This is the change you'll feel first, and it needs no setup at all. Spinning up git worktrees to juggle several branches or reviews at once is a core moflo workflow, so we made it a default: when moflo starts in a checkout that belongs to a repo with worktrees in play, your durable learnings converge across every one of them on their own.

The trick is that there's nothing to configure because there's nothing to coordinate. Every linked worktree of a repo shares a single .git common directory, so moflo derives the shared store at a path they all compute the same way — <git-common-dir>/moflo/durable.db — and they converge on it without a registry, a config key, or a daemon talking to another daemon. (Conductor builds on git worktrees under the hood, so its workspaces are covered for free.)

At the start of every session, before its daemon even comes up, each checkout does two quick things: it flushes its own new learnings up to the shared store, and it seeds any siblings' learnings back down. The net effect is that all your worktrees converge on the union of everything any of them has learned. Every durable write also writes through to the shared store as it happens, so you're never more than a session behind. Structural and ephemeral data stays in each checkout's own local database, exactly where it belongs.

Two deliberate guardrails make this safe to have on by default:

  • It only wakes up for worktrees. A plain single checkout derives nothing and writes nothing — behaving byte-for-byte as it did before. The check is a couple of filesystem stats, so if you're not using worktrees you pay nothing.
  • It can't trip the index hazard. The derived store is a dedicated durable.db, never a full moflo.db — so only the safe durable slice ever travels, and the HNSW divergence problem from the previous section simply can't arise.

Want it somewhere else, or off? Set memory.durable_path to override the location (or to converge separate clones, which don't share a .git), or memory.worktree_sharing: false to turn it off entirely. The /memory-worktree skill walks you through verifying it, moving it, or opting out — but for the common case, you do nothing.

Beyond one machine — teams and your other laptops

Worktrees share a disk. The moment learnings need to leave this machine, the destination is the same whether it's a teammate or your own laptop: a file that travels with the repo. "Just me on two machines" isn't a separate mechanism — it's a team of one. So both use one path, and the learnings live in the repo, reviewed like any other change. This mode uses a text format instead of a binary one:

flo memory team-export     # writes .moflo/shared/learnings.jsonl and makes it git-trackable
git add .moflo/shared/learnings.jsonl && git commit && git push

The file is JSONL — one learning per line, sorted by namespace and key — precisely because a committed file needs to be diff-reviewable and merge-friendly. You can see in a pull request exactly what knowledge a teammate is adding. Embeddings are omitted (384 floats per row would make every diff unreadable); they're regenerated automatically on import. After anyone runs git pull, their next session auto-imports the merged file, and each entry is stamped with its git author so you can see where a given learning came from. No one has to remember to run an import command — it happens at session start.

One small piece of plumbing worth mentioning, because it's a classic git gotcha: you can't re-include a file with ! if a parent directory is ignored. So team-export rewrites your .gitignore from a blanket .moflo/ to .moflo/* plus !/.moflo/shared/ — ignoring the machine-local database while tracking just the shared artifact.

You still have to remember to export before you commit, though — and that's exactly the kind of thing people forget. So the /memory-team skill also installs a pre-commit hook that re-runs team-export and stages the artifact on every commit (only when it actually changed, so there's no empty-diff churn). The learnings you picked up while building a change get committed with that change, automatically — the artifact is never stale in a pull request.

Prefer to keep your own cross-machine learnings out of git? There's a private alternative that skips the repo entirely — a portable binary artifact you carry through a synced folder, USB stick, or scp:

flo memory sync --to   ~/Dropbox/moflo-learnings.db   # on machine A
flo memory sync --from ~/Dropbox/moflo-learnings.db   # on machine B

Because it's a plain SQLite file, it carries the embeddings verbatim — no CPU burned re-computing vectors on the other side — and the same INSERT OR IGNORE merge means running it twice, or in both directions, never duplicates or clobbers anything. The trade-off is that it isn't diffable and isn't reviewed; it's just for you.

The fast-start bonus: whole-DB snapshots

Everything above shares the durable slice. But there's a related problem it doesn't solve: on a large repo, a brand-new workspace with an empty database has to do a full cold reindex on its first session — parse all your code, re-embed everything through the ONNX model — which can take minutes before search works at all.

So there's one more, complementary tool: a whole-database snapshot.

flo memory backup  --to   ~/moflo-snapshot.db    # consistent snapshot of the entire DB
flo memory restore --from ~/moflo-snapshot.db    # hydrate a fresh workspace instantly

Or set memory.hydrate_from and a fresh workspace seeds itself automatically at session start. Because the snapshot already contains the structural namespaces and their embeddings, the expensive first-pass reindex is skipped entirely — search works on session one.

"But wait," you might say, "didn't you just spend three paragraphs explaining why sharing the whole database is dangerous?" The distinction is live versus one-time. The trap is two running daemons writing one shared file, where their in-memory indexes drift apart. A snapshot restore is a one-time copy into an empty workspace that then owns its own copy — there's no second concurrent daemon, so there's no index to diverge. Any branch drift in the structural data simply self-heals on the next incremental reindex. Different operation, no hazard.

How the pieces fit together

Automatic worktree sharing, the team artifact, and snapshot hydration aren't competitors — they compose, and they run in a deliberate order at session start, all before the daemon comes up:

1. hydrate    → restore a whole-DB snapshot (only if the local DB is empty)
2. purge      → drop ephemeral namespaces from the restored copy
3. durable    → flush + seed the durable slice against the shared store
4. team       → import the git-tracked team artifact
5. daemon     → starts, builds its index over everything above

A new workspace bootstraps instantly from a snapshot, then the durable-slice sync keeps its learnings continuously converged with everyone else's — your other worktrees, your other machines, your team. Snapshot is the fast cold-start; durable-slice sharing is the ongoing convergence. They never fight, because every step is purely additive — INSERT OR IGNORE on that same unique key.

Setup, and what's on by default

Here's the whole surface, from "do nothing" to "wire up a team":

Your situation What you do What travels
Worktrees / Conductor on one machine Nothing — it's automatic A shared SQLite store in .git
A team on one repo — or just you across several machines Run /memory-team once A git-tracked JSONL file
A fresh workspace on a huge repo Set memory.hydrate_from A whole-DB snapshot (one-time)

You shouldn't have to hand-assemble paths, config keys, and git hooks, so two guided skills own the setup. /memory-team configures the git-tracked artifact and installs the pre-commit hook — for a whole team or just your own machines. /memory-worktree is for the automatic case's edge cases: confirm it's working, move the shared store, extend it to separate clones, or opt out.

Under the hood it's all one memory: block — every line optional, and every line off unless you set it:

memory:
  # worktree_sharing: false                            # opt OUT of the automatic worktree default
  # durable_path: ~/.moflo-shared/team-learnings.db    # override the auto store / share across separate clones
  # team_artifact: .moflo/shared/learnings.jsonl        # git-tracked JSONL — team, or you across machines
  # hydrate_from:  /path/to/moflo-snapshot.db           # seed a fresh workspace from a snapshot

Which is to say: almost everything here is off until you ask for it. The single exception is durable sharing across worktrees, on by default because worktrees are core to how people use moflo — and even that is conservative, waking up only when a repo actually has worktrees and staying byte-for-byte silent otherwise. And if you ever wire it up the risky way — pointing a config at a shared full database rather than a dedicated durable-only store — flo doctor -c shared-db catches it and warns you before it bites (it also reports when automatic worktree sharing is active, so you can always see what's on).

Why this matters

The value of an assistant that learns is only as good as its reach. A lesson MoFlo learned on Monday, on your desktop, in your main checkout, isn't worth much if it's invisible on Tuesday, on your laptop, in a review worktree — or to the teammate who hits the exact same gotcha next week.

These changes make the learning portable without making it dangerous. The trick was refusing to treat "the database" as one thing: share the small, durable, human-meaningful slice freely; let the large, branch-specific, machine-generated part stay put and rebuild itself. That one distinction is what lets your worktrees, your machines, and your whole team all get smarter about a project together — while search stays fast and correct on every one of them. And for the worktree case, the workflow moflo leans on hardest, it now happens without you lifting a finger.

Teach the project once. Have it remembered everywhere it runs.
