§1Executive summary

What we built and why, in one screen.

moflo stores everything it learns about a project in a local SQLite database (.moflo/moflo.db), which is gitignored. That means every fresh git worktree, every Conductor workspace, every new machine, and every teammate starts with an empty learnings namespace — accumulated knowledge is silently lost.

The fix is built on one key insight: only a thin slice of the DB is worth sharing — the user-authored learnings and knowledge namespaces. Everything else (code maps, guidance chunks, run state) is derived or ephemeral and rebuilds cheaply, so it stays local. Sharing the whole DB would make each daemon's in-memory search index diverge.

We shipped three sharing mechanisms (chosen by topology), plus documentation and a safety check:

Same machine

Shared durable store

memory.durable_path — worktrees / Conductor workspaces seed + write-through to one durable-only DB.
One user, many machines

Portable artifact

flo memory sync --to/--from — carry a durable SQLite file via a synced folder.
A team

Git-tracked JSONL

flo memory team-export → commit .moflo/shared/learnings.jsonl; teammates import-merge on pull.
Safety & docs

Doctor + guidance

flo doctor -c shared-db warns against whole-DB sharing; a shipped guidance doc explains it all.

§2The problem

Why durable knowledge evaporates across installations.

  • Origin: a Conductor support request — .moflo/ is gitignored, so Conductor workspaces (and git worktrees) lose the learnings namespace on every rotation.
  • Generalized: the same gap hits any topology where moflo runs from more than one checkout/machine — solo multi-machine, and teams on a shared repo.
  • The trap to avoid: live-sharing one moflo.db between two running daemons. node:sqlite + WAL prevents row loss, but each daemon builds its own in-memory HNSW index — a write from install A never updates install B's index, so search silently returns stale results, and structural namespaces churn across branches.

A distinct, safe operation (planned — #1244): restoring a one-time whole-DB snapshot to seed a fresh/empty workspace is not the trap above. Each workspace then owns its own copy (no second concurrent daemon → no index divergence), and the incremental reindex reconciles branch drift. It's the right tool for fast cold-start: an empty DB forces a full parse + ONNX re-embed of all code/patterns/tests/guidance on the first session — minutes on a large repo — which a snapshot skips. "Never share the whole DB" is specifically about live concurrent sharing; snapshot-restore composes with the durable-slice sync (which keeps learnings converged afterward).

§3The core model: three data classes

The decision that makes everything else safe — share only the durable slice.

ClassNamespacesShared?Why
Durablelearnings, knowledgeYesUser-authored, slow to recreate, identical across checkouts
Structuralcode-map, guidance chunks, tests, patternsNoBranch-specific; rebuilds cheaply from the indexers
Ephemeraltasklist, hive-mind, epic stateNoPer-run scratch; meaningless in another install

All three mechanisms move the same durable slice and dedupe on the existing UNIQUE(namespace, key) constraint via INSERT OR IGNORE, so combining them is conflict-free and idempotent.

§4The three mechanisms at a glance

Pick by who needs the learnings — not by what's easiest to wire.

Why JSONL for teams but SQLite elsewhere: a committed team file must be diff-reviewable and merge-friendly; embeddings (384 floats/row) would make every diff unreadable, so they're dropped and regenerated. For the SQLite modes, carrying embeddings verbatim avoids a lossy/expensive re-embed.

#1232Durable-only shared store — memory.durable_path

Same-machine worktrees & Conductor workspaces share one learnings DB. PR #1236 · commit c39568a35

High level: set memory.durable_path to a dedicated store. On session-start each checkout seeds from it (pulls siblings' learnings) and flushes to it; every durable write also writes through. Structural namespaces stay in the local .moflo/moflo.db.

src/cli/services/durable-sync.ts (new) memory/entries-write.ts (write-through) config: memory.durable_path launcher block 3e-1232
Detailed design & implementation

Mechanism

  • resolveDurablePath() — env MOFLO_DURABLE_PATH > memory.durable_path; relative → project root; a self-reference guard disables it if it would alias the local DB.
  • seedDurableFromShared() / flushDurableToShared() — both are a single call to the existing cherryPickLearningsFromLegacy() primitive with source/target swapped (INSERT OR IGNORE, embeddings carried, no new SQL).
  • syncDurableAtSessionStart() — flush then seed → the union of durable rows on both sides. Runs before the daemon spawns so freshly-seeded rows are present when it builds its HNSW index.
  • writeThroughDurable() — after a durable write lands locally, propagate to the shared store. Best-effort and fully guarded.

Key cross-platform detail

Path identity reuses normalizeProjectRoot (realpath symlinks + Windows case-fold, ref #1145) so both sides of any comparison fold identically — including a stableAbsolute helper for not-yet-created targets.

Review fixes folded in

  • Correctness: writeThroughDurable's resolveDurablePath/loadMofloConfig call was moved inside the try/catch — a config-load throw could otherwise fail a write that had already persisted.
  • Hot path: the launcher block got a cheap pre-gate (env or an uncommented durable_path: in moflo.yaml) so solo users load no module and parse no yaml each session.

Default-off guarantee

durable_path defaults undefined; with it unset, behavior is byte-identical to before (feature-off tests + the pre-gate prove it).

#1233Portable artifact — flo memory sync

One user carries learnings between their own machines. PR #1237 · commit 4a5028816

High level: flo memory sync --to <file> exports the durable slice to a portable SQLite artifact (keep it in Dropbox/iCloud or copy by hand); --from <file> merges it back. Idempotent; embeddings ride along.

src/cli/commands/memory.ts (sync subcommand) reuses cherryPickLearningsFromLegacy + expandHome (~ expansion)
Detailed design & implementation

Why a SQLite artifact (not JSONL)

It carries embeddings verbatim (no lossy re-embed), is conflict-free/idempotent (INSERT OR IGNORE), and adds zero new serialization format — it reuses the same DB→DB primitive as restore-learnings. The diffable/JSONL artifact was deliberately deferred to the team story (#1234), where text-diffability is actually load-bearing.

Notable discovery

The pre-existing flo memory export/import commands call memory_export/memory_import MCP tools that don't exist — those commands are effectively dead. sync deliberately uses the working cherry-pick primitive instead. (Flagged for a separate cleanup.)

Review fixes folded in

  • Tilde expansion: path.resolve('~/x') doesn't expand ~; added expandHome handling ~/, ~\, and bare ~ (Rule #1).
  • Honest reporting: self-reference (artifact aliases the local DB) and a TOCTOU 0/0 result now warn instead of falsely reporting "merged N".
  • --from prints a restart / rebuild-index nudge so merged rows become searchable.

#1234Git-tracked team artifact (JSONL)

A whole team accumulates each other's learnings about a repo. PR #1238 · commit bc9f5ed82

High level: flo memory team-export writes .moflo/shared/learnings.jsonl (and makes it git-trackable); a teammate commits it; everyone else's session-start import-merges it after git pull. Diffable, reviewable, merge-friendly.

src/cli/services/team-artifact-sync.ts (new) commands: team-export / team-import config: memory.team_artifact launcher block 3e-1234
Detailed design & implementation

Format & semantics

  • JSONL — one entry per line, sorted by (namespace, key) → line-local git diffs, conflict-free merges for distinct keys. Embeddings omitted (regenerated on import).
  • First-write-wins both ways via UNIQUE(namespace, key): re-export never rewrites a teammate's line; import is INSERT OR IGNORE.
  • Provenance — each entry is stamped with git author + hostname, retained in the imported row's metadata.

The .gitignore re-include problem

Git can't re-include a file if a parent dir is excluded, so .moflo/ + !.moflo/shared/... does not work. ensureSharedArtifactTracked() rewrites the ignore to .moflo/* + !/.moflo/shared/ (gitignore globs always use /, even on Windows).

Review fixes folded in (a 3-agent Opus review)

  • Reuse: extracted a shared DURABLE_INSERT_OR_IGNORE_SQL + hasMemoryEntriesTable() into cherry-pick-learnings.ts so the two durable-row writers can't drift columns.
  • Silent-drop: INSERT OR IGNORE swallows any constraint violation — out-of-CHECK-set type is now coerced to semantic, non-durable namespaces are skipped before the insert, and timestamps never bind null.
  • gitignore: strips a bare .moflo/ even when .moflo/* co-exists; never injects a spurious rule for a custom artifact outside .moflo/.
  • Hot path: launcher pre-gate + a single transaction around the import (one fsync, not per-row).
  • Rule #1: a stray NUL byte in the source (made git treat the file as binary) was caught and replaced with a runtime String.fromCharCode(0).

§5Supporting fixes (surfaced during review)

Pre-existing issues that had to be cleared to land the epic on green CI.

#1242 — hono override

Raised the transitive hono override past a high-severity advisory (>=4.12.25) so the required Lint & Security npm audit gate passes repo-wide. Zero source impact.

#1239 — daemon-rpc flake

A test pinned a random port (30000+rand) that collided under parallel CI. Fixed by binding port: 0 (OS-assigned) and reading the real port back — a small, behavior-identical change to daemon-dashboard.ts.

#1240 — graph-analyzer flake

A test derived its temp dir from Date.now(), colliding in the same millisecond so files leaked between cases. Fixed with mkdtemp (OS-guaranteed unique dir). Verified 10× green.

Common thread: for test isolation, let the OS allocate the unique resource (mkdtemp for dirs, listen(0) for ports) rather than constructing a "probably-unique" name.

§6Quality & review process

How confidence was built before landing.

  • Per-story: each PR ran /flo-simplify (a tiered reviewer; #1234 hit the DEEP tier → 3 parallel Opus reviewers for reuse / quality / efficiency).
  • Epic-wide: three parallel deep reviewers across all branches — architecture & cohesion, Rule #1 cross-platform, and regression & consumer blast-radius — plus a full test-suite run on each branch.
  • Regression caught by the full suite: team-artifact-sync.ts was missing from the moflo.db writer-audit whitelist (a #1054/#1055 governance lint); added with a daemon-offline classification.
Architectural-review verdict (summary)

Mechanically sound — inter-PR conflicts were trivial keep-both (config fields, command-array appends), resolved during the merge. No semantic landmines. The one standing design note (non-blocking): three transfer surfaces over one cherry-pick core — worth converging on a single flo memory sync verb + a shared durable-transfer module eventually. The protected swarm/hive-mind/MCP surface was untouched.

§10How it landed

Stabilize-then-merge, past branch protection.

The landing sequence & rationale

main requires 1 review + 5 green CI checks. With no second reviewer available and the epic author being the sole committer, merges used an admin override for the review gate only — never to bypass red checks. CI was made genuinely green first:

  1. Stabilize: land the hono bump (clears the repo-wide npm audit gate) and both flaky-test fixes before the epic, so the required Tests/Security checks pass on the epic PRs.
  2. Merge in order: #1236 → #1237 → #1238 → #1241, rebasing each branch onto the updated main before merging.
  3. #1238 conflicts (moflo-config.ts, memory.ts) resolved keep-both; #1234's local isDurableNamespace collapsed onto #1232's exported one. That merge passed the full Tests suite in CI.

Integrated main verified: build clean + integration suites green (sync/team/durable coexist, writer-audit whitelist intact, doctor check live). All four story issues auto-closed by their PRs.
