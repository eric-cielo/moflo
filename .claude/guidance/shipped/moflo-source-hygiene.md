# Source Code Hygiene Rules

**Purpose:** Prevent stale artifacts, phantom directories, and architectural confusion in the moflo monorepo. Apply these rules whenever creating, editing, or reviewing source files.

---

## All Source Files Are TypeScript

**Every source file under `src/` MUST be a `.ts` file.** There are no legitimate hand-written `.js` files in source directories. If you encounter a `.js` file under `src/`, it is either:

| Situation | Action |
|-----------|--------|
| A stale build artifact (`.js`, `.js.map`, `.d.ts`, `.d.ts.map`) | Delete it immediately — it was accidentally committed |
| A hand-written `.js` file | Convert it to `.ts` or delete if redundant |
| A file in a `dist/` directory | Leave it — that's compiled output |

**Never edit `.js` files under `src/`.** If you find yourself editing one, stop and find the corresponding `.ts` source. The `.js` file is either stale output or a problem that needs fixing.

---

## Single Source Tree Under `src/cli/`

**The entire moflo source lives at `src/cli/`** after the workspace-collapse epic ([#586](https://github.com/eric-cielo/moflo/issues/586) / [ADR-0001](../../docs/adr/0001-collapse-moflo-workspace-packages.md)). There is no longer a `src/modules/<pkg>/` workspace tree, no per-package `tsconfig.json`/`package.json`, and no `@moflo/<pkg>` workspace-package specifiers.

| Pattern | Status |
|---------|--------|
| `import { Foo } from '../bar.js'` | CORRECT — same-tree relative import |
| `import { Foo } from '@moflo/neural'` | WRONG — that workspace package no longer exists |
| `import { Foo } from '../../../../modules/<pkg>/...'` | WRONG — fixed-depth pre-collapse paths (banned by ESLint per [feedback_no_fixed_depth_paths.md](../../docs/adr/0001-collapse-moflo-workspace-packages.md)) |
| `import { Foo } from 'src/modules/...'` | WRONG — that tree was deleted in #602 |

The drift guard at `src/cli/__tests__/services/published-package-drift-guard.test.ts` enforces that no `@moflo/*` bare specifiers reappear in source.

---

## moflo Publishes ONE npm Package: `moflo`

The directories under `src/cli/` (e.g., `commands/`, `neural/`, `shared/`, `hooks/`, `memory/`, `swarm/`, `spells/`, `guidance/`, `aidefence/`, `embeddings/`) are internal subsystems that get bundled into the single published package. They are NOT independently published to npm.

There is no `@moflo/cli`, `@moflo/neural`, `@moflo/shared`, etc. Those workspace package names existed pre-collapse and are dead.

When you need to add a feature, place it in the existing subsystem that owns the concern. Don't create a new top-level subsystem under `src/cli/` without explicit approval.

---

## Canonical Locations

**Each type of file has exactly ONE canonical home.** Do not create parallel implementations.

| Artifact | Canonical Location | Notes |
|----------|-------------------|-------|
| MCP tool definitions | `src/cli/mcp-tools/` | All MCP tools live here |
| CLI commands | `src/cli/commands/` | All `flo <command>` handlers |
| Shared types/utilities | `src/cli/shared/` | Cross-subsystem shared code |
| Tests | `src/cli/__tests__/` or `tests/` | Never alongside source |
| Build output | `dist/src/cli/` (repo root) | Single tsc build, not per-package |
| Guidance docs | `.claude/guidance/shipped/` | Prefix with `moflo-` |

**If you find duplicate implementations** of the same functionality in different directories, one is legacy. Determine which is authoritative (usually the one with more features, tests, and active imports) and remove the other.

---

## Build Artifacts Must Not Be Committed

**Never commit files from `dist/` directories to git.** The `.gitignore` excludes them. If you see `.js`, `.d.ts`, or `.map` files under source control, remove them.

Compiled output is generated by `npm run build` (single `tsc` run from repo root). `tsconfig.json` specifies `outDir: "./dist"` and the build emits to `dist/src/cli/`.

---

## Async-Eligible Operations Must Be Async By Default

**Use the async API for any operation that has one, unless a documented reason forces sync.** Sync IO and busy-waits block the event loop — on a hot path that pins a CPU core, stalls timers, and turns a "best-effort" failure into a hang. Issue #854 burned 5s of CPU to a `while (Date.now() < end)` busy-wait inside an already-async launcher.

| Operation | Use | Avoid |
|-----------|-----|-------|
| File IO inside an `async` function | `fs/promises` (`readFile`, `copyFile`, `mkdir`) | `*Sync` variants |
| Delays | `await new Promise(r => setTimeout(r, ms))` | `while (Date.now() < end)` busy-wait |
| Spawning long-running children | `await new Promise` around `spawn`/`execFile` | `execFileSync`/`spawnSync` (Windows timeouts orphan children — #744) |

**Sync is OK only when:** the function cannot be `async` (top-level config reader before any await; certain CommonJS hooks); the IO is microscopic on a known-fast path (one `existsSync` probe, one tiny `readFileSync`); or conversion churn outweighs the benefit. Document the reason in a one-line comment.

**Rule of thumb:** if anything in your call stack already `await`s, every IO/delay below it must be async too — mixing sync IO inside an async function is a smell.

## When Creating New Files

Before creating any new source file, verify:

1. **Does this file already exist?** Search for existing implementations first
2. **Is this the right directory?** Check the canonical locations table above
3. **Is this necessary?** Prefer editing an existing file over creating a new one
4. **Will it ship?** If it needs to reach consumers, verify the root `package.json` `files` array includes its compiled output
